## 1. 운용 철학

### 4대 원칙

1. **LLM 최소·구조화 먼저** — RAG·벡터·템플릿·알고리즘으로 풀 수 있으면 LLM 안 쓴다
2. **기본은 `gpt-4o-mini`** — 상위 모델은 증거 기반으로만 선택
3. **캐시 적극 활용** — 반복 쿼리는 무조건 캐시, 히트율 60% 목표
4. **유효성은 결정적 검증, 품질은 사용자 신호** — 추상적 "품질 점수" 사용 금지

### 절대 하지 말 것

- 가상의 "품질 점수"로 자동 재시도 (측정 불가능한 추상화)
- 모든 요청을 4o로 (비용 6배, 품질 차이 거의 없음)
- Batch API를 동기 UX에 사용 (24h SLA 위반)
- 캐시 없이 프로덕션 운영 (비용·지연 모두 악화)

## 5. Fallback 전략 (0.7 threshold 대체)

추상적 "품질 점수 0.7"은 구현 불가능. **3개 구체 트리거**로 교체.

### Trigger A: 결정적 검증기 (자동 재시도)

mini 응답이 아래 조건 중 하나라도 실패 시 → **자동으로 4o 재시도 + 로그**

| 검증 | 조건 | 구현 |
|------|------|------|
| JSON 스키마 | PLAN_TEMPLATE_SCHEMA valid? | Zod/Ajv 검증 |
| 필수 필드 | dates, venues, activities 존재? | 필드 체크 |
| 이동 시간 합리성 | 서울→부산 10분 같은 오류? | 지리 거리 계산 |
| 금지 표현 | 의료법 위반 표현 포함? | 키워드 블록리스트 |
| 길이 임계 | 응답 < 300 chars or > 10,000? | 단순 카운트 |
| 언어 일치 | 요청 언어 (JP) 로 응답? | langdetect |

**예상 발생률**: 5-10% (mini 응답 대부분 유효)
**비용 증가**: 해당 건만 4o 재시도 → 전체 평균 +50% (단, 해당 요청만)

### Trigger B: 사용자 신호 (명시적 업그레이드)

플랜 결과 하단에 버튼:
```
┌────────────────────────────────────┐
│ 💎 "더 정교한 플랜 만들어줘"       │
│    (500 포인트 차감 · 고급 AI)     │
└────────────────────────────────────┘
```

- 사용자가 품질 판단 주체
- 포인트 차감 → 비용 회수
- UX 명확, 자동화 복잡도 없음

### Trigger C: 샘플링 (장기 품질 튜닝)

- 전체 요청의 **5%**를 4o에도 백그라운드 호출
- 내부 품질 대시보드에 누적
- 임계 (예: 4o가 80% 이상 사용자 재생성 감소) 넘으면 mini→4o 상향 결정

### Config 매핑

```
ai.model.plan_fallback              = "gpt-4o"
ai.model.plan_fallback_trigger      = "validator_or_user_signal"
                                      # enum: "never" / "validator_only"
                                      #       / "user_signal_only"
                                      #       / "validator_or_user_signal"
ai.model.sampling_rate              = 0.05  # Trigger C, 5%
ai.model.user_upgrade_cost_points   = 500
```

## 6. 캐시 전략

### 캐시 레이어 3단

```
[요청] ──▶ L1 앱 Response Cache (Redis, TTL 24h)
            │ Miss
            ▼
           L2 OpenAI Prompt Cache (자동, 시스템 프롬프트)
            │ Miss
            ▼
           L3 RAG 검색 결과 Cache (TTL 1h, 축제·명인 DB)
            │ Miss
            ▼
          [OpenAI API 호출]
```

### 캐시 키

```typescript
const cacheKey = hash({
  prompt_normalized: normalize(userPrompt),    // 소문자·공백 정규화
  rag_doc_ids: topKDocIds.sort(),              // 검색된 문서 ID
  user_locale: user.locale,                    // jp/en/kr
  config_version: config.version,              // 설정 변경 시 자동 무효화
  model: "gpt-4o-mini",                        // 모델 바뀌면 무효화
});
```

### 무효화 규칙

| 트리거 | 무효화 대상 |
|--------|-----------|
| 축제 데이터 갱신 | `festival_id=*` 포함 캐시 |
| 사용자 설정 변경 | `user_id=X` 포함 캐시 |
| Config 버전 변경 | 이전 버전 키 자동 만료 |
| TTL 만료 (24h) | 자동 |

### 유스케이스별 적중률 목표

| 유스케이스 | 목표 히트율 | 근거 |
|-----------|----------|------|
| 플랜 생성 | 30~50% | 여행 트렌드 반복 (서울 3박 4일 등) |
| 채팅 Q&A | 10~20% | FAQ 스타일 질문 |
| 이미지 번역 | 60~80% | 메뉴판·간판 반복 텍스트 |
| 아이스브레이커 | 40~60% | 비슷한 Mate 프로필 |
| 음성 통역 | 5~10% | 상황 의존성 높음 |

**평균 목표**: **60%**

### Config 매핑

```
ai.cache.enabled         = true
ai.cache.ttl_hours       = 24
ai.cache.max_size_mb     = 1000
ai.cache.rag_ttl_hours   = 1
```

## 7. Batch API 전환 기준

### 언제 켜나?

Batch API는 **24시간 SLA** 비동기. 현재는 전부 동기 UX. M5 이후 백그라운드 작업 생기면 활용.

### 후보 유스케이스 (M5+)

- [ ] 야간 플랜 품질 재평가 샘플링 (Trigger C)
- [ ] Venue DB 일괄 다국어 번역 (신규 축제 등록 시)
- [ ] 장기 유저 행동 요약 리포트
- [ ] A/B 실험 결과 분석
- [ ] 공급자 업로드 상품 1차 AI 심사

### 전환 결정 기준

다음 3개 중 2개 이상 충족 시 활성:
1. 동기 요청 아닌 비동기 사용 사례 월 10,000건 이상
2. 해당 유스케이스 비용 절감 > 50% 예상
3. 24h SLA 허용 가능 (사용자 대기 없음)

### Config 매핑

```
ai.batch_api.enabled   = false
ai.batch_api.use_cases = []  # M5 이후 배열에 추가
```

## 8. Break-even 분석

### 매출 vs AI 비용

| 지표 | 값 |
|------|-----|
| FestiPoint 팩 평균 판매 (1 사용자/년) | ¥6,000 ≈ ₩55,000 |
| 10K MAU 연 매출 (팩 판매만) | ~5.5억원 |
| 10K MAU AI 비용 (캐시 60%) | ~$576/년 ≈ 80만원 |
| **AI 비용률** | **~0.15%** |

### 결론

AI 비용은 **사업 리스크 아님**. 100K MAU에서도 0.15% 수준. 오히려 **"LLM 더 써도 됨"** 여지 있음.

하지만 원칙상 최소화가 맞음:
- 속도 (mini 더 빠름)
- 단순성 (구조화 > 복잡한 프롬프트)
- 확장성 (요금 급락 시 유리)

## 9. KPI 모니터링

### 추적 지표

| 카테고리 | 지표 | 목표 |
|---------|------|------|
| 사용량 | AI 호출 수 / 일·월 | MAU 증가율과 선형 |
| 사용량 | 평균 토큰 (model × use case) | 안정 |
| 캐시 | 적중률 | ≥ 60% |
| 캐시 | 무효화 빈도 | 과도한 무효화 없음 |
| Fallback | Trigger A (validator 실패) 비율 | ≤ 10% |
| Fallback | Trigger B (사용자 upgrade) 비율 | ≤ 15% |
| Fallback | Trigger C (sampling) 비율 | 고정 5% |
| 에러 | timeout / rate limit / invalid | < 1% |
| 사용자 | 플랜 저장률 | ≥ 60% |
| 사용자 | 재생성 클릭률 | ≤ 15% |

### 알람 임계값

| 조건 | 조치 |
|------|------|
| 월 AI 비용 > 예산 * 1.2 | 알림 |
| 에러율 > 5% | 즉시 조사 |
| 캐시 적중률 < 20% | 캐시 키 전략 재검토 |
| Fallback A > 20% | mini → 4o 상향 고려 |
| Fallback B > 30% | 사용자 불만 신호, 모델 재검토 |

## 10. 업그레이드 결정 체크리스트

`mini` → `4o` 기본값 상향 결정 시:

- [ ] Fallback Trigger A (validator 실패) > 15%
- [ ] Fallback Trigger B (user upgrade) > 30%
- [ ] 샘플링 (C) 에서 4o가 사용자 재생성 감소 80%+ 입증
- [ ] 현재 월 비용 < 예산 50% (여유)
- [ ] 3개월 이상 데이터 축적 (단기 변동 배제)

**3개 이상 Yes** → 상향 (config만 변경, 코드 수정 없음)
**그 외** → 유지
