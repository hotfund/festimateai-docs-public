## 1. 핵심 원칙

### 1-1. 3층 정보 아키텍처

일본 여행자의 멘탈 모델에 맞춰 카테고리를 3층으로 분류:

```
┌──────────────────────────────────────────────────────────┐
│  Layer 1: Discovery (발견) — 여행 축을 만드는 것            │
│  🎪 축제 · 🏆 명인명장 · 🏯 템플스테이 · 🌿 정원             │
│  위치: /search (메인)                                       │
│  공통점: "여기 가는 게 여행 목적"                           │
├──────────────────────────────────────────────────────────┤
│  Layer 2: Specialized Research (전문 리서치) ⭐ 메인 수익원   │
│  💄 K-뷰티                                                 │
│  위치: /beauty (별도 페이지, P1 우선 설계)                   │
│  홈에서 프라임 노출 필수 (히어로 근처)                       │
│  공통점: "고관여 · 긴 의사결정 · 전용 검색축 필요"             │
├──────────────────────────────────────────────────────────┤
│  Layer 3: Contextual Utility (상황 유틸)                    │
│  🛏 KSleep · 🛍 Shopping · 🍜 KFood                        │
│  위치: /search 사이드 아이콘 (부가) + AI Guide 자동 추천      │
│  공통점: "여행 중 상황 보고 결정, 단독 탐색 대상 아님"         │
└──────────────────────────────────────────────────────────┘
```

### 1-2. 왜 이 구조인가

| 관점 | 효과 |
|------|------|
| 지도 노이즈 제거 | 5만 개 식당·숙소 마커로 덮이지 않음 |
| 여행자 멘탈 모델 일치 | "어디 갈까" vs "이거 얼마야" vs "지금 뭐 먹지" 분리 |
| AI 역할 선명 | Utility = AI 담당 = PLAYBOOK "AI handles logistics" 실현 |
| BM 정합 | **K-뷰티 = 메인 수익원** (Session 2 격상) = 전용 UX 투자 근거 명확 |
| 콘텐츠 볼륨 관리 | Utility 데이터는 AI 컨텍스트로 해결, 지도에 안 뿌림 |

### 1-3. K-뷰티 격상의 의미 ⭐ Session 2

CLI 세션 2에서 확정: **K-뷰티가 FestiMate의 메인 수익원.**

| 변화 | 내용 |
|------|------|
| 홈 배치 | "3-카드 (Festival·Mate·Beauty)" → **K뷰티 대표 섹션 + 3-카드** 2단 구조 |
| /beauty 페이지 우선순위 | "추후 설계" → **P1 우선 설계** |
| 개발 리소스 배분 | Discovery 카테고리 균등 대신 K-뷰티에 가중치 |
| 법무·규제 | 의료법 §56, 외국인환자유치업, 의료분쟁조정법 등 선제 검토 필수 |
| 공급자 온보딩 | 클리닉 파트너십이 Master·Templestay보다 앞 |

**이 문서는 `/search` 설계에 초점.** `/beauty` 페이지 상세 설계는 별도 `docs/BEAUTY_PAGE_DESIGN.md` (신규 작성 필요).

## 2. 확정된 설계 결정 (11개)

### 2-1. 페이지 구조 — `/search` 하나로 통합

- 기존 `/explore` 제거 또는 `/search` 리디렉션
- 키워드 + 필터 + 지도 + Utility 부가 = 모두 한 페이지
- 이유: "검색"과 "탐색" 혼동 제거

### 2-2. 모바일 UX — Airbnb 스타일

- 지도 전체화면 + 바텀시트 카드 (슬라이드업)
- 네이티브 앱 느낌, 2026년 지도 검색 UX 정석
- 일본인 유저 이미 익숙 (Airbnb·마이리얼트립 등)

### 2-3. 지도 라이브러리 — Google Maps + 카카오 보조

- Main: Google Maps (일본인 친숙도 최우선)
- 행정구역 경계: TopoJSON 오버레이
- 지하철·KTX 경로: 카카오맵 API 보조 데이터
- 3개월 후 Mapbox 전환 검토 여지 (비용·커스텀 필요 시)

### 2-4. 지도 기본 노출 요소

- **지하철 노선**: 기본 ON (토글 아님)
- **대한민국 영토**: 제주·울릉 포함
- **카테고리 마커**: DESIGN.md §Color 색상 준수
  - festival `#EF4444` · master `#8B5CF6` · templestay `#14B8A6` · garden `#10B981`
  - beauty `#3B82F6` · food `#F59E0B` · sleep `#06B6D4` · shopping `#EC4899`

### 2-5. 카테고리 선택 — 상위 다중 선택

- Discovery 4개 (축제·명인·템플·정원) 체크박스 형식 다중 선택
- 사용자 시나리오: "5월 축제 + 명인 + 서울 강남 압구정" 조합
- **기본 상태**: 🎪 축제만 ON, 나머지 OFF

### 2-6. 월별 필터

- 축제·정원 선택 시 자동 노출, 명인·템플 선택 시 숨김
- 최대 **2개월 동시 선택** (예: 3월 + 4월)
- "더 정확히" 버튼 → 날짜 범위 선택기 (datepicker) 전환 옵션

### 2-7. 지역 필터 — 드릴다운 + 단일 선택

```
Level 1: 시·도 (17개 + 전체)
Level 2: 시·군·구 (선택한 시 내부, 예: 서울 → 25개 구)
Level 3: 상권 배지 (주요 구만, 예: 강남구 → 압구정·신사·역삼)
```

- 상권 배지 적용 구: 강남·중구·마포·종로 등 주요 5-10개
- 단일 선택 (카테고리와 달리 다중 아님)
- "전체" 리셋 버튼

### 2-8. Utility 레이어 — 사이드 아이콘

지도 우측에 3개 아이콘:
```
🛏 KSleep
🛍 Shopping
🍜 KFood
```

- 기본 OFF, 클릭 시 해당 마커 레이어 추가
- Shopping은 District 단위 (명동·강남·홍대·성수·동대문·가로수길·이태원·인사동)

### 2-9. 교통 정보 — 3층 역할 분담

- **/search 페이지**: 지하철 노선 기본 노출 (더 이상 X)
- **장소 상세**: "🚄 서울역 KTX 50분" 뱃지
- **AI Guide**: 플랜 생성 시 교통 자동 최적화
- **Mate 페이지** (별도 설계): 히어로 메시지 *"メイトは現地の地理と交通をその場で解決します"*

### 2-10. 축제 전용 — 캘린더 뷰 토글

- `[지도 뷰] [캘린더 뷰]` 토글 버튼
- 캘린더에서 "5월" 클릭 → 지도에 5월 축제만 하이라이트
- 지도에서 축제 클릭 → 캘린더에 해당 날짜 하이라이트

### 2-11. K-뷰티 분리 ⭐ Session 2 갱신

**메인 수익원이지만 `/search`에서 분리된 이유**:
- 검색 축이 다름 (시술종류·가격·리뷰·언어지원이 중심, 지역보다 상위)
- 사용자 행동이 다름 ("브라우징" 아닌 "리서치", 평균 세션 시간 3-5배 길음)
- 법무 요건 다름 (의료법 §56, 외국인환자유치업)

**그럼에도 홈에서 프라임 노출**:
- 홈 히어로 바로 아래 **K-뷰티 대표 섹션** 배치 (Session 2 결정)
- "인기 시술 TOP 5", "한국 VIP 피부과 10선" 등 콘텐츠 섹션
- Discovery 3-카드보다 **상위 배치**

**/beauty 페이지** 별도 설계는 `docs/BEAUTY_PAGE_DESIGN.md` (신규 문서 필요, P1에 작성).

## 5. 인터랙션 패턴

### 5-1. 마커 클릭 플로우

```
마커 클릭 → 바텀시트 확장 → 해당 카드 하이라이트
       ↓
   카드에 [❤️ 위시] [➕ 플랜담기] [📍 상세보기] 버튼
```

### 5-2. 위시리스트 · 플랜 담기

- 모든 카드에 ❤️ (위시) + ➕ (플랜) 버튼
- 위시리스트: 로그인 필요, 영구 저장
- 플랜 담기: 세션 로컬 (비로그인 가능), AI Guide로 전송

### 5-3. 상단 sticky 카운터

```
[플랜에 담긴 N개] → AI에게 보내기 →
```

담긴 여러 장소를 AI Guide 페이지로 한번에 전송.

### 5-4. URL 공유 (Shareable)

모든 필터 상태 URL 쿼리로 직렬화:
```
/search?cat=festival,master&month=5,6&region=seoul&district=gangnam
```

소셜·LINE·카톡 공유 시 필터 그대로 살아있음.

## 7. 미결 사항 (Open Decisions)

다음 세션 또는 구현 시 결정:

1. **클러스터링 구체 방식** — 줌 레벨 임계값, 시각적 표시
2. **위시리스트 데이터 모델** — DB 스키마, 동기화 전략
3. **공유 URL 직렬화 포맷** — 정확한 쿼리 파라미터 이름
4. **텍스트 검색 + 필터 AND/OR** — 기본 AND인지 토글 허용인지
5. **결과 수 제한** — 명인 193명 전체 표시 vs 페이지네이션
6. **빈 상태 UI 디자인** — 필터 적용 후 결과 0건일 때
7. **KTX 뱃지 정확한 위치·포맷** — 카드 상단 vs 상세만
8. **다국어 장소명 병기** — 한일영 동시 표시 vs 언어 설정 따라
9. **Shopping District 프로필 콘텐츠** — 각 지구 소개 작성 필요 (10개)
10. **상권 배지 데이터 소스** — 수동 큐레이션 vs API
11. **K-뷰티 수익 모델** ⭐ Session 2 — referral fee(건당 정액) vs commission(%) vs 하이브리드
12. **의료 상담 신청 폼 수집 정보** ⭐ Session 2 — 법무 검토 전제로 최소 필드 결정

## 8. 관련 admin 설정 (ADMIN_CONFIGURABLE_PARAMS)

이 페이지에서 runtime에 변경 가능해야 할 값:

```
search.default_active_categories  → ["festival"]  (기본 ON 카테고리)
```
