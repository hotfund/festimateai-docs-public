# Mate 투어제안서 만들기 페이지 설계

> **작성**: 2026-04-20
> **위상**: Mate Offering 생성 페이지 SSOT — `/mate/offering/new` 구현 기준
> **선행 문서**: `PLAN_TEMPLATE_SCHEMA.md` §5-4 (MateOffering 스키마), `AI_PLANNER_PAGE_DESIGN.md` §5·§16~17 (Mate 5필드·Level 시스템·항목 태그), `MEMBERSHIP_TIERS.md` §3 (Mate 등급별 Offering 한도)
> **현행 구현**: 없음 (신규)
> **설계 목표**: 승인된 Mate가 "Looking for Tourist" 투어 제안서를 Level 1~3으로 단계적으로 작성하는 도구

---

## 1. 핵심 원칙

### 1-1. 진입 장벽 최소화
- **Level 1 (5필드, 5분)** 만 채워도 즉시 게시 가능
- Level 2·3은 "더 완성도 높은 제안서 = 더 많은 매칭 기회"로 유도 (강제 아님)
- 처음 작성하는 Rookie도 10분 안에 첫 Offering 게시 가능해야 함

### 1-2. 같은 컴포넌트, 다른 역할
- Level 3 Day-by-Day 편집기 = Tourist AI Planner 결과 화면과 동일 컴포넌트 (`PlanGrid`)
- 단, Mate 편집기는 **수동 편집 + AI 도움** 모드 추가
- 같은 JSON 구조 (`TravelPlan.days`) 사용 → 비교 뷰(View 3) 코드 재사용 가능

### 1-3. AI는 보조, Mate가 주인공
- AI는 Level 3 작성 시 "뼈대 초안" 생성만 담당
- 장소 선택·시간 배분·특기사항은 Mate가 직접 편집
- "AI가 다 만들어준 제안서"가 되면 안 됨 → Mate 전문성이 차별점

## 9. 특정 MateRequest에 응답하는 흐름

Matching 페이지에서 Tourist 요청 카드를 보고 "이 여행자에게 제안하기":

```
Tourist 요청 카드 → [제안하기]
  → /mate/offering/new?request_id=req_xxx
  → Level 1 폼 자동 채움:
      dates: Tourist 여행 날짜
      region: Tourist 희망 지역
      pair_format: Tourist 희망 페어
  → Mate가 가격·소개·Level 2/3 채움
  → 게시 시 response_to_request_id 연결
  → Tourist에게 알림: "새 제안서가 도착했어요"
```

이 경우 Offering이 해당 Tourist에게 **우선 노출** (매칭 가능성 UP).

## 11. 미결 사항

1. **AI 초안 생성 모델**: gpt-4o-mini로 충분한가, gpt-4o 필요한가?
2. **장소 검색 데이터소스**: 자체 DB 장소만? 네이버지도 API 외부 검색?
3. **일본어 제목 필수 여부**: Mate가 한국어로 써도 자동 번역? 아니면 직접 입력 필수?
4. **만료 정책**: `available_dates` 종료일 = 자동 expired? 아니면 별도 설정?
5. **Offering 수정**: 게시 후 수정 가능? 매칭 중인 Offering 수정은?
