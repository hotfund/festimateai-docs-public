# Matching Page 설계 (AI Matching Page Design)

> **버전**: v0.2
> **작성**: 2026-04-20
> **업데이트**: 2026-04-20
> **위상**: AI Matching 페이지 SSOT — `/ai/matching` 구현 기준
> **선행 문서**: `AI_PLANNER_PAGE_DESIGN.md` §7·§10·§16-3 (매칭 정책), `MEMBERSHIP_TIERS.md` §3 (Mate 등급), `AI_TERMINOLOGY.md` (용어)
> **현행 구현**: `/mate` 스캐폴드 (목록만, 매칭 기능 없음)
> **설계 목표**: Tourist가 AI 플랜을 가지고 와서 맞는 동반자를 찾아 "공동 여행 설계"를 시작하는 화면

---

## 1. 핵심 원칙

### 1-1. 상호 존중 매칭 (Mutual Interest)
- Tourist도 Mate도 **먼저 관심 표시** 가능 (쌍방향, `matching.initiating_mode = "bidirectional"`)
- 매칭 성사 조건: **상호 ❤️** — 한쪽만은 안 됨 (`matching.approval_mode = "mutual"`)
- 정보 개방은 3단계 점진적 언락 (Tier 1→2→3)

### 1-2. 정보 개방 3단계 (Tier Unlock)

| Tier | 조건 | 공개 정보 |
|------|------|---------|
| **Tier 1** (공개) | 매칭 페이지 진입만 해도 | Level 배지, 지역, 전문 분야, 언어 수준, 가격대, 후기 수 |
| **Tier 2** (상호 ❤️) | 양쪽 관심 표시 완료 | 실명, 연락처 (카카오/LINE), 포트폴리오 사진, 상세 후기 |
| **Tier 3** (예약 확정) | 결제·예약 완료 | 정확한 만남 장소, 전화번호, 긴급 연락처 |

### 1-3. v1 티저 모드
- **v1.0**: 카드 렌더링 + 잠금 오버레이 + Mate 사전등록 유도
- **v1.5**: `matching.v1_teaser_mode = false` → 실제 매칭 활성화 (Mate 10명+ 확보 후)

### 1-4. 매칭 범위 정의 (스코프)

| 케이스 | 처리 방침 |
|--------|---------|
| 일본인 → 한국인 메이트 | ✅ 핵심 설계. 주 마케팅 타겟 |
| 일본인 → 일본인 여행파트너 | ✅ 旅仲間 모드로 지원 |
| 비일본 외국인 → 한국인 | 🕐 v2+ (아키텍처는 열어둠) |
| 한국인 → 한국인 | 시스템 허용, 마케팅 안 함 |

**메이트 역할 정의**: 메이트는 유료 가이드일 수도, 무료 친구 같은 동행일 수도 있다.
역할은 사전에 고정하지 않는다. 메이트 프로필에 보상 기대를 표시하고, 매칭 후 둘이 협의한다.

```

├──────────────────────────────────────┤
│ Zone 2: 매칭 모드 탭 + 필터 바       │
```

| 필터 | UI | 옵션 |

## 6. Zone 3 — Mate 카드

### 6-1. 카드 구조 (Tier 1 공개 정보)

```
┌─────────────────────────────────────┐
│ 👤 [프로필 사진 - 60x60 원형]       │
│ [PRO] 이름 (익명화: "지수 M.")      │
│ ⭐ 4.9 · 127 ガイド完了              │
│                                     │
│ 📍 서울 강남·홍대 전문               │
│ 💬 日本語 N1 · 한국어 원어민         │
│ 🎯 K-뷰티·카페·트렌드 패션           │
│ 💰 ¥8,000 ~ ¥12,000 / セッション   │
│ 👥 FF 페어 전문                      │
│                                     │
│ 📝 "일본어과 4학년. 압구정·가로수길  │
│     단골 많아요 ✨"                  │
│ (140자 Offering 소개 미리보기)       │
│                                     │
│      [💜 気になる]  [📋 詳細]         │
└─────────────────────────────────────┘
```

### 6-2. Level 배지 표시

| 배지 | 이름 | 색상 | 의미 |
|------|------|------|------|
| `[PRO]` | Pro Guide | 🟣 보라 | 관리자 심사 통과, 최고 등급 |
| `[VER]` | Verified | 🔵 파랑 | 5회+ 완료, 4.5+ 평점 |
| `[RKY]` | Rookie | 🟢 초록 | 신규 승인, 0~4회 |

Tourist에게 설명 문구 제공:
> 「PRO = 30回以上 · VER = 5回以上 · RKY = 新メイト(高コスパ)」

### 6-2-1. 메이트 카드 — 보상 타입 표시 추가

기존 카드 `💰 ¥8,000 ~ ¥12,000` 라인을 보상 형태에 따라 다르게 표시:

| 보상 형태 | 카드 표시 |
|---------|---------|
| 유료 (`paid`) | `💰 有料 ¥8,000~¥12,000 / セッション` |
| 무료 (`free`) | `🤝 無料 (交流目的)` |
| 협의 (`negotiable`) | `💬 要相談 (食事など)` |

메이트가 프로필 등록 시 선택. 변경 가능.

### 6-3. 카드 정렬 기준

기본 정렬: AI 플랜 유사도 + 레벨 가중치

```
score = (plan_match_score × 0.6) + (level_weight × 0.2) + (rating × 0.2)
```

- plan_match_score: 지역·날짜·전문분야 일치도
- level_weight: PRO=1.0, VER=0.8, RKY=0.5
- rating: 0~1 정규화

수동 정렬 옵션: 평점 높은 순 / 가격 낮은 순 / 최신 등록 순

### 6-4. 旅仲間 카드 (여행파트너 모드)

```
┌─────────────────────────────────────┐
│ 🇯🇵 [프로필 사진 - 60x60 원형]      │
│ ゆり · 東京  ·  여성 · 20대         │
│                                     │
│ 📅 4/25~27 (2박3일) 서울            │
│ 🎯 K-뷰티·カフェ巡り·コスメショッピング│
│ 👥 FF 동반 선호                     │
│ 💸 費用: 均等割り                   │
│                                     │
│ 📝 "初めての韓国！クリニックと明洞   │
│     一緒に行ける方探してます🌸"      │
│                                     │
│   [一緒に行きたい ❤️]  [詳細を見る] │
└─────────────────────────────────────┘
```

**旅仲間 카드 Tier 1 공개 정보**: 이름(닉네임), 출신국/도시, 성별, 연령대, 여행 날짜, 여행 도시, 관심사, 비용 분담 방식, 한 줄 소개
**Tier 2 (상호 ❤️)**: LINE ID, 실명, 여행 플랜 상세, 숙소 정보 (공유 의향 있는 경우)

## 7. 매칭 액션 플로우

### 7-1. Tourist 관심 표시 (❤️)

```
Tourist → [💜 気になる] 클릭
  → 비로그인: "ログインして続ける" 모달
  → 로그인됨: 관심 저장 (DB)
     → Mate에게 알림: "새 관심 표시가 왔어요 (플랜 요약 포함)"
     → Tourist 화면: "✓ 気になる済み" 변경
```

### 7-2. Mate 맞관심 → Tier 2 언락

```
Mate → [気になる] 클릭 (Mate 대시보드에서)
  → 매칭 성사!
  → 양쪽에 알림: "マッチング成立 🎉"
  → Tier 2 정보 언락:
     Tourist에게: Mate 실명, 카카오/LINE ID, 포트폴리오 사진 10장, 상세 후기
     Mate에게: Tourist 실명, 연락처, 상세 플랜
  → 채팅방 생성 (별도 `/chat/[match_id]` 라우트)
```

### 7-4. 旅仲間 매칭 플로우

```
Tourist A → [一緒に行きたい ❤️] 클릭
  → 비로그인: 로그인 모달
  → 로그인됨: 관심 저장
     → Tourist B에게 알림: "旅仲間候補が現れました！"
     → Tourist B가 맞관심 → 매칭 성사!
     → Tier 2 언락: LINE ID, 상세 플랜 공유
     → 채팅방 생성 (Tourist-Tourist 채팅)
```

旅仲間 매칭은 메이트 매칭과 동일한 상호 ❤️ 방식. 차이점:
- 양쪽 다 Tourist (한쪽이 Mate가 아님)
- 보상/결제 없음 (각자 부담 또는 협의)
- Tier 3 (예약 확정) 단계 없음 — Tier 2 언락 후 자유롭게 진행

### 7-3. 예약 확정 → Tier 3 언락

```
채팅에서 날짜·장소 합의
  → [예약 확정] 버튼
  → 결제 (Phase 2 이후, v1은 무료)
  → Tier 3 언락: 전화번호, 정확한 만남 장소
  → 상태: BOOKED
```

## 8. v1 티저 모드 UX

`matching.v1_teaser_mode = true` 상태:

### 8-1. 카드 오버레이

```
┌─────────────────────────────────────┐
│ 👤 [흐린 프로필]                    │
│ [VER] 지수 M.  ⭐ 4.7 · 43ガイド    │
│ 📍 서울  💬 日本語 N2               │
│                                     │
│  ┌───────────────────────────────┐  │
│  │  🔒  準備中                   │  │
│  │  先行登録でマッチング開始時に  │  │
│  │  最初に通知を受け取れます      │  │
│  └───────────────────────────────┘  │
│                                     │
│      [先行登録する →]               │
└─────────────────────────────────────┘
```

- 카드 Tier 1 정보는 **흐리게(opacity 50%)** 표시 (존재는 보임, 클릭 불가)
- 오버레이는 [X] 없음 (닫기 불가)
- 「先行登録」 클릭 → 이메일 수집 모달

### 8-2. 사전등록 모달

```
┌─────────────────────────────────────┐
│ 🎉 メイトマッチング先行登録         │
│                                     │
│ メイトが準備できたらすぐお知らせ！  │
│                                     │
│ [メールアドレス             ]        │
│ [どのメイトに興味ありますか?] (chip)│
│  ☐ K-뷰티  ☐ 축제  ☐ 맛집  ☐ 기타 │
│                                     │
│ [登録する →]                        │
└─────────────────────────────────────┘
```

- 이메일 + 관심 분야 2가지만 수집
- 등록 후: "登録完了! 準備ができたらメールでお知らせします 🙌"
- DB: `waitlist` 테이블 (email, interests, plan_id, created_at)

### 8-3. 현황 배너 (Zone 3 상단)

```
┌─────────────────────────────────────┐
│ 🤝 現在 XX名 のメイトが準備中       │
│ 先行登録で優先マッチングをゲット！  │
└─────────────────────────────────────┘
```

숫자는 실제 Mate 승인 대기자 수 또는 사전등록자 수 표시.

## 15. 미결 사항

1. **채팅 구현 방식** — Supabase Realtime vs Sendbird vs 자체 polling? 비용·복잡도 트레이드오프
2. **LINE 알림** — 일본 사용자는 LINE 선호. LINE Notify 또는 LINE Messaging API 연동 여부
3. **Mate 프로필 사진 가이드라인** — 얼굴 공개 필수인가? (Tier 1=흐림, Tier 2=선명)
4. **익명화 수준** — Tier 1에서 "지수 M." vs "M 메이트" vs 완전 익명?
5. **Tourist 후기 시스템** — Mate만 후기 받는가, Tourist도 받는가?
6. **Mate 카드 최소 수** — 0명일 때 빈 화면 vs "준비 중" 배너만?
7. **AI 매칭 점수 공개 여부** — "XX% 일치" 숫자를 사용자에게 보여줄지?
8. **무한 스크롤 vs 페이지네이션** — 모바일 UX 결정
9. **/ai/matching vs /mate** — 현행 `/mate` 라우트 대체인가, 공존인가?
